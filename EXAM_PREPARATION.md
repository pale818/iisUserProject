```properties

jwt.refresh-token-expiry-ms=1209600000 # 14 days (default: 7 days)
```

## WSDL / SOAP

**"Show me the WSDL"**
Open in browser: `http://localhost:8080/ws/users.wsdl` — no code change.

**"Search by email too"**
One line in `UserSoapService.java` — extend the XPath expression:
```java
" or contains(translate(email,'ABCDEFGHIJKLMNOPQRSTUVWXYZ','abcdefghijklmnopqrstuvwxyz'),'" + term + "')"
```

**"Add a new field to the SOAP request"**
Three files:
1. `users-soap.xsd` — add the new `xs:element`
2. `SearchUsersRequest.java` — add the field + getter/setter
3. `UserSoapService.java` — use the new field in the logic

**"What is WSDL?"**
WSDL (Web Services Description Language) is an XML file that describes a SOAP service — what operations exist, what input/output they expect, and where the service is located. It is the contract between the client and the server. In this project it is auto-generated by Spring WS from `users-soap.xsd`.

**"What is the difference between SOAP and REST?"**
- REST uses plain HTTP + JSON, is simple and stateless
- SOAP uses XML envelopes with a strict structure, has a formal contract (WSDL), better for enterprise/legacy systems
- REST is more common today; SOAP is still used in banking, government, integrations

## JWT

**"What happens when the access token expires?"**
The `JwtFilter` calls `jwtService.isTokenValid(token)` on every request. If expired, the filter does nothing, the request reaches Spring Security unauthenticated, and a 401 is returned. The client must call `POST /auth/refresh` with the refresh token to get a new access token.

**"Where is the role stored?"**
In the `app_users` table in H2, in the `role` column. It is re-read from the database on every request inside `AppUserDetailsService.loadUserByUsername()`.

**"Add a third role"**
Three places:
1. Seed a new user in `IisUsersProjectApplication.java` with the new role string
2. Add the role rules in `SecurityConfig.java` inside `authorizeHttpRequests`
3. That's it — `AppUserDetailsService` already converts any role string to a `GrantedAuthority`

**"Why stateless JWT instead of sessions?"**
Sessions store state on the server (memory or DB) — every server instance must share it. JWT is stateless — all needed info is in the token itself, verified with a secret key. Scales better, no shared session store needed.

**"Difference between access token and refresh token?"**
- Access token: short-lived (15 min), used on every API call in the `Authorization` header
- Refresh token: long-lived (7 days), stored in the DB, used only to get a new access token when the old one expires
- Short access token expiry limits damage if stolen — it stops working quickly

## Filtering / XPath

**"How does the XPath filtering work?"**
`XmlGenerationService` fetches ReqRes users and builds an XML document in memory. `UserSoapService` parses that XML into a DOM, then runs an XPath expression against it. The expression finds all `<user>` nodes where `firstName` or `lastName` contains the search term (case-insensitive via `translate()`). Matching nodes are serialized back to XML and returned in the SOAP response.

**"Why XPath and not a database query?"**
The data source here is an XML file generated from the ReqRes REST API, not the local database. XPath is the standard way to query XML documents — the same way SQL queries a database.

**"Change filtering to starts-with instead of contains"**
One line in `UserSoapService.java` — replace `contains(...)` with `starts-with(...)`:
```java
String expression = "//user[starts-with(translate(firstName,...),'" + term + "')" +
        " or starts-with(translate(lastName,...),'" + term + "')]";
```

**"Add fuzzy/approximate matching"**
XPath 1.0 does not support fuzzy matching natively. You would drop the XPath approach and loop through all `<user>` nodes in Java, applying a distance algorithm (e.g. Levenshtein). The XML parsing and response building stays the same — only the matching logic changes.

## GraphQL

**"What is the difference between GraphQL and REST?"**
REST returns a fixed response shape — the server decides what fields come back. GraphQL lets the client ask for exactly the fields it needs. Reduces over-fetching (getting too much data) and under-fetching (needing multiple calls).

**"Add an update mutation"**
Two files:
1. `schema.graphqls` — add:
```graphql
type Mutation {
    updateUser(id: ID!, input: UserInput!): User
}
```
2. `UserGraphQLController.java` — add:
```java
@MutationMapping
public User updateUser(@Argument Long id, @Argument Map<String, String> input) {
    return userService.updateLocalUser(id, ...);
}
```

**"Why @Controller and not @RestController for GraphQL?"**
Spring GraphQL routes requests through its own handler, not the standard MVC response pipeline. `@RestController` adds `@ResponseBody` which tells Spring to serialize the return value directly as HTTP response body — that conflicts with GraphQL's own serialization. `@Controller` lets Spring GraphQL handle the response.

## gRPC / Protobuf

**"What is the difference between gRPC and REST?"**
- REST: HTTP/1.1 + JSON — human readable, simple, widely supported
- gRPC: HTTP/2 + binary Protobuf — faster, smaller payloads, strongly typed, better for service-to-service communication

**"Why Protobuf?"**
Protobuf is a binary serialization format. It is faster to serialize/deserialize than JSON, produces smaller payloads, and the schema (`weather.proto`) auto-generates type-safe Java classes. The contract is enforced at compile time.

**"Add a new field to the weather response"**
Three files, then `mvn clean compile`:
1. `weather.proto` — add the field, e.g. `string humidity = 5;`
2. `WeatherFetchService.java` — extract the new value from DHMZ XML
3. `WeatherGrpcService.java` — set the new field on the response builder

## XML / JSON Validation

**"Add a new required field to the schema"**
Two files:
- `user.xsd` — add `<xs:element name="newField" type="xs:string"/>`
- `user-schema.json` — add `"newField": { "type": "string" }` and add `"newField"` to the `required` array

**"What happens with invalid input?"**
`XmlValidationService` validates against `user.xsd` using `javax.xml.validation`. If invalid, it returns a list of error strings. The controller returns HTTP 400 with `{"errors": [...]}`. The user is not saved.

**"What is the difference between XSD and JSON Schema?"**
Both describe the structure and constraints of a data format. XSD is for XML, JSON Schema is for JSON. In this project both validate the same user data shape but in their respective formats.

## Adding a new API endpoint

Pattern is always the same — three files, in this order:

**1. `UserRepository.java`** — add Spring Data method if needed:
```java
List<User> findByFirstNameContainingIgnoreCase(String name);
```
Spring Data generates the SQL automatically from the method name.

**2. `UserService.java`** — add the business logic:
```java
public List<User> searchByName(String name) {
    return userRepository.findByFirstNameContainingIgnoreCase(name);
}
```

**3. `UserController.java`** — add the endpoint:
```java
@GetMapping("/search")
public List<User> search(@RequestParam String name) {
    return userService.searchByName(name);
}
```

Security is already covered — all `/api/**` endpoints are protected automatically by `SecurityConfig`.

## General architecture questions

**"How would you add a PostgreSQL database instead of H2?"**
Change three lines in `application.properties` — the JPA/entity code does not change at all:
```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/mydb
spring.datasource.driver-class-name=org.postgresql.Driver
spring.datasource.username=postgres
spring.datasource.password=yourpassword
```
And swap the H2 dependency for the PostgreSQL driver in `pom.xml`.

**"Why H2 for this project?"**
No installation needed, runs embedded in the JVM, perfect for development and demo. The same JPA code works with any relational database by changing the datasource config.

**"What is the role of Spring Security here?"**
Every HTTP request passes through `JwtFilter` before reaching any controller. The filter reads the `Authorization` header, validates the JWT, loads the user role from the DB, and puts it in `SecurityContextHolder`. Spring Security then checks that role against the rules defined in `SecurityConfig` before allowing or blocking the request.

**"What is the API switch in the GUI?"**
The frontend can toggle between calling `GET /api/users` (local H2 database) and `GET /api/users/public` (live ReqRes public API). Both return the same user shape so the table renders the same way. This demonstrates the interoperability theme of the project.
